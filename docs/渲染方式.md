这是一个非常经典的前端架构问题。这四个概念都是**渲染网页的方式**，核心区别在于：**HTML 代码在什么时候、什么地方生成**。

为方便理解，先建立一个简单模型：

- **构建时**：你 `npm run build` 的时候
- **请求时**：用户访问页面的时候
- **运行时**：浏览器加载 JS 后

下面逐一讲解。

---

### 1. CSR (Client-Side Rendering) - 客户端渲染

**流程：**
1. 浏览器请求 → 返回一个几乎空的 HTML（只有 `<div id="root"></div>`）
2. 浏览器下载 JS
3. JS 执行 → 获取数据 → 动态生成 DOM 内容

**特点：**
- 首屏慢（白屏时间长）
- SEO 差（爬虫拿不到内容）
- 适合后台系统、需登录的强交互应用

> 典型框架：React + CRA / Vite 默认模式

---

### 2. SSR (Server-Side Rendering) - 服务端渲染

**流程：**
1. 浏览器请求 → **服务端实时运行框架代码**（如 React 的 `renderToString`）
2. 服务端生成完整 HTML 返回
3. 浏览器显示 HTML，然后下载 JS 绑定事件（注水，hydration）

**特点：**
- 首屏快（HTML 已经包含内容）
- SEO 好
- 服务端压力大（每次请求都重新渲染）
- 常见方案：Next.js 的 `getServerSideProps` / Nuxt

> 典型场景：新闻、电商商品页、论坛帖子

---

### 3. SSG (Static Site Generation) - 静态站点生成

**流程：**
1. **构建时**就生成所有页面的静态 HTML
2. 部署到 CDN
3. 用户请求 → 直接返回已生成好的 HTML

**特点：**
- 性能极快（纯静态，无服务端计算）
- SEO 完美
- 内容必须是**构建时已知**的（如博客、文档）
- 无法处理用户个性化内容

> 典型框架：Next.js 的 `getStaticProps` / Hugo / Astro

---

### 4. ISR (Incremental Static Regeneration) - 增量静态再生

**原理（以 Next.js 为例）：**
- 和 SSG 一样先生成静态页
- 设置一个 `revalidate` 时间（如 60 秒）
- 用户访问时：
  - 先返回**旧的缓存页面**
  - 同时在后台触发重新生成
  - 下一次请求看到新内容

**特点：**
- 兼顾 SSG 的性能 + 内容可更新
- 不用全量重新构建（增量更新）
- 适合**数据频繁变化但不要求实时**的场景：电商库存、博客文章更新

> 典型配置：`getStaticProps` + `revalidate`

---

## 核心对比表

| 维度 | CSR | SSR | SSG | ISR |
|------|-----|-----|-----|-----|
| HTML 生成时机 | 浏览器运行时 | 每次请求时 | 构建时 | 构建时 + 定期/触发更新 |
| 首屏速度 | 慢 | 较快 | 极快 | 极快（先缓存后更新） |
| SEO 友好 | 差 | 好 | 最好 | 最好 |
| 服务端压力 | 低 | 高 | 无 | 低（仅按需更新） |
| 实时性 | 高（需单独请求数据） | 高 | 低 | 中（有缓存延迟） |
| 适合场景 | 后台、复杂交互 | 实时内容、个性化 | 文档、博客、落地页 | 动态内容 + 高性能要求 |

---

## 选择建议

- **内容几乎不变**（如帮助文档） → **SSG**
- **内容经常变，需要 SEO，可接受少量延迟** → **ISR**
- **内容实时且个性化**（如用户主页、购物车） → **SSR**
- **不需要 SEO，完全内部系统** → **CSR**

如果你用 **Next.js**，实际项目中往往混合使用：  
- 博客用 SSG  
- 商品详情用 ISR  
- 用户订单用 SSR  
- 个人设置用 CSR
