大公司的 Git 开发流程，核心是通过"**规范的流程 + 自动化的工具 + 严格的制度**"来保障大规模、多人协作下的代码质量和系统稳定性。虽然具体的实践细节会因公司而异，但底层逻辑和主流模式已经非常成熟。

下面我会先梳理出核心的工作流模式，然后用一个贯穿场景带你走一遍完整的流程。

### 🏛️ 两种主流工作流模式

目前，大公司普遍采用以下两种工作流模式或其变种，它们的核心区别在于对"发布"过程的管控。

| 特性 | **Git Flow** (适合有固定发布周期的项目) | **Trunk-Based Development (TBD)** (适合需要持续交付/部署的互联网项目) |
| :--- | :--- | :--- |
| **核心分支** | `master` (生产) 和 `develop` (集成) 长期并存 | 只有一个 `master/main` 主干分支，所有工作都基于它 |
| **工作方式** | 功能分支 (`feature/*`) 从 `develop` 切出，完成后合并回 `develop` | 极短生命周期的功能分支，要求开发者**每天至少**向主干合并一次代码 |
| **发布流程** | 从 `develop` 拉出 `release/*` 分支进行发布前的最终测试和修复 | 从 `master` 直接拉出 `release` 分支进行发布，或者直接从 `master` 部署 |
| **优点** | 流程清晰，各阶段职责分明，对大型复杂项目管控力强 | 集成更频繁，冲突少，能快速发现和解决问题，实现快速迭代 |
| **适用场景** | 传统软件、需要多版本并行的项目、客户定制化项目 | 互联网SaaS应用、微服务、需要每日/每周多次发布的敏捷团队 |

现在，越来越多追求效率和敏捷性的团队倾向于采用更轻量的 **TBD 模型**。下面的流程也会主要基于这种模式来介绍。

### 💡 全流程解析：从需求到上线

以一个具体的需求为例，比如开发一个"用户上传头像"的功能，来看看整个协作流程。

#### 1. 功能开发：从主干创建分支

流程开始于从最新的 `master` 分支创建一个新的功能分支。分支命名通常会关联需求ID，便于追溯。

```bash
# 拉取最新的master分支代码
git checkout master
git pull origin master

# 创建并切换到功能分支
git checkout -b feature/123-user-avatar-upload
```

#### 2. 日常开发：小步提交，勤推送

在功能分支上，开发者应**频繁地**进行"原子化"提交，即每个提交只做一件事。提交信息需要遵循统一规范，便于自动化生成日志和定位问题。

```bash
# 开发完成后，添加并提交
git add .
git commit -m "feat(profile): 实现用户头像上传的UI组件"

# 再次提交另一个逻辑点
git commit -m "feat(profile): 集成云存储SDK用于头像上传"

# 将本地的多个提交推送到远程仓库
git push origin feature/123-user-avatar-upload
```
> **规范示例**：`<类型>(范围): <描述>`，例如 `feat`: 新功能， `fix`: 修复bug， `docs`: 文档更新等。

#### 3. 代码审查 (Code Review)：PR/MR 机制

功能开发完成后，开发者不会直接合并，而是创建一个 **Pull Request (PR)** 或 **Merge Request (MR)**，请求将代码合并到 `master` 分支。

*   **发起PR**：在GitLab/GitHub等平台上创建PR，清晰地描述需求、实现方案和测试情况，并至少指定一名同事作为审查者。
*   **自动化检查**：PR创建后，会自动触发CI/CD流水线，运行单元测试、代码风格检查、安全扫描等。**只有所有自动化检查都通过，合并按钮才会亮起**。
*   **人工审查**：审查者关注代码逻辑、设计、可维护性等方面，并在PR下进行评论和讨论。通常需要至少 **1-2 人批准**才能进行下一步。

#### 4. 合并与集成

PR通过审查且所有检查通过后，拥有合并权限的人（或由系统自动）执行合并。合并时，**强烈推荐使用 "Squash and merge"** 策略，将功能分支上所有琐碎的提交压缩成一个有意义的提交，合并到 `master` 分支，保持主干历史的干净整洁。

```bash
# 相当于在PR界面点击“合并”按钮，背后执行了类似的操作
git checkout master
git merge --squash feature/123-user-avatar-upload
git commit -m "feat: 支持用户头像上传功能 (#123)"
```

#### 5. 发布与部署

`master` 分支的代码始终保持可部署状态。

*   **测试环境**：当功能合并到 `master` 后，CI/CD流水线会自动将其部署到测试环境，供QA团队验证。
*   **生产发布**：当一批功能准备好发布时，团队会从当前的 `master` 分支拉出一个 `release` 分支（例如 `release/v1.2.0`）。在这个分支上只进行bug修复和发布准备工作，不再添加新功能。测试通过后，`release` 分支被合并回 `master`，并打上一个版本标签。

```bash
# 发布时打标签
git tag -a v1.2.0 -m "发布版本v1.2.0，新增用户头像上传功能"
git push origin v1.2.0
```

#### 6. 线上紧急修复 (Hotfix)

如果在生产环境发现了严重Bug，需要立即响应。Hotfix流程允许从 `master` 分支直接拉出一个 `hotfix` 分支进行修复，以避免干扰主干上的新功能开发。

```bash
# 从master分支创建hotfix分支
git checkout master
git checkout -b hotfix/critical-login-error

# 修复问题 -> 提交 -> 创建PR -> 审查通过后合并回master
# 合并后同样打一个补丁版本标签，如 v1.2.1
```

### ⚙️ 保障这一切运转的"护栏"

如此严谨的流程能在大公司里顺利执行，离不开一系列自动化规则的保障，这些规则通常在 Git 服务端（如 GitLab/GitHub）配置：

*   **分支保护**：`master`、`release/*` 等核心分支会被设置为"受保护"，禁止任何人直接推送代码。
*   **强制PR**：所有对受保护分支的更改都必须通过PR，且必须经过指定人数的审阅者批准。
*   **检查状态**：PR必须关联需求工作项，并且所有CI检查（测试、构建等）必须通过，合并按钮才可用。
*   **自动清理**：PR合并后，系统会自动删除源分支，保持仓库整洁。

### 💎 总结

大公司的 Git 流程是一个系统工程。它通过 **Trunk-Based Development** 等模式实现高效协作，通过 **详细的提交规范** 保证代码历史清晰可读，并通过 **PR 代码审查** 和 **自动化CI检查** 建立质量防线，最后用 **分支保护和强制规则** 将这一切固化下来。

这是一个很好的问题。简单直接的答案是：**PR（Pull Request）和 MR（Merge Request）在本质上是一回事，没有功能上的区别。** 它们指代的都是**请求将代码从一个分支合并到另一个分支**的这个协作机制。

唯一的区别在于**不同Git托管平台的习惯叫法**：

- **PR (Pull Request)**：是 **GitHub** 和 **Bitbucket** 使用的术语。
- **MR (Merge Request)**：是 **GitLab** 使用的术语。

为了让你理解得更透彻，可以拆开看一下这个词的由来，也能解释为什么会有两种叫法。

### 为什么会有两个名字？

这反映了两种略微不同的**思考角度**，但最终操作和目的完全一样。

1.  **PR (Pull Request) - “请你拉取我的代码”**
    -   **视角**：发起者的角度。
    -   **逻辑**：我（在特性分支上）做了一些提交。现在，我请求你（仓库维护者）**从我的分支拉取（Pull）** 这些更改到你的主分支。
    -   **关键词**：**拉取**。

2.  **MR (Merge Request) - “请求合并我的代码”**
    -   **视角**：动作本身的角度。
    -   **逻辑**：我创建了一个请求，内容是希望**将我的分支合并（Merge）** 到目标分支。
    -   **关键词**：**合并**。

> **一句话总结**：就像“土豆”和“马铃薯”，名字不同，指的都是同一样东西。在 GitHub 上你创建的就是 PR，在 GitLab 上创建的就是 MR。

### 核心流程完全一致

无论是在 GitHub 上创建一个 PR，还是在 GitLab 上创建一个 MR，你都会经历完全相同的流程：

1.  **发起**：从你的功能分支，向目标分支（如 `main`）发起一个合并请求。
2.  **描述**：填写标题、描述，说明这个更改解决了什么问题。
3.  **审查**：指定同事作为审查者（Reviewer），他们会查看你的代码改动并发表评论。
4.  **自动化**：系统会自动运行测试、代码风格检查等CI/CD流程。
5.  **讨论与修改**：根据反馈，你可能需要继续向你的功能分支推送新的提交。这些提交会自动更新到这个 PR/MR 中。
6.  **合并**：在获得足够数量的批准且所有自动化检查通过后，由有权限的人点击“合并”按钮，将你的代码正式合入目标分支。

### 一张表看清对应关系

| 概念 | GitHub (叫 PR) | GitLab (叫 MR) |
| :--- | :--- | :--- |
| **核心动作** | Pull Request | Merge Request |
| **发起请求的人** | 贡献者 (Contributor) | 作者 (Author) |
| **审查代码的人** | 审查者 (Reviewer) | 维护者/审查者 (Maintainer/Reviewer) |
| **最终合并** | 点击 **"Merge pull request"** | 点击 **"Merge"** |

### 不同公司/团队的习惯

-   **互联网大厂（如 Google, Meta）**：它们内部会有自己定制的Git系统，但**理念上更接近 MR**，因为他们会使用类似 **“Merge Request”** 的术语来描述这个流程。
-   **开源项目**：绝大多数托管在 GitHub 上，所以大家习惯叫 **PR**。提过 PR 是参与开源贡献的标准方式。
-   **企业内部 GitLab**：团队内部就会统一叫 **MR**。

### 总结

-   **面试或对外交流时**：说 **PR** 更通用，因为 GitHub 的生态更广。
-   **在 GitLab 环境下工作时**：入乡随俗，说 **MR**。
-   **本质上**：**没有任何区别**。当你听到别人说“提一个 PR”或“发一个 MR”，指的都是“发起一个代码合并请求”这个动作。

以后你只需要记住：**PR = MR**，只是不同平台的不同叫法而已。
