### 推荐的 Git Flow（安全、标准）

你的分支结构（`master`、`version`、`test`）已经很接近标准 Git Flow，只需要微调合并方向：

| 分支 | 用途 |
|------|------|
| `master` | **线上生产代码**，永远与线上环境一致 |
| `version`（如 `release/2.0.0`） | **版本集成分支**，用于本版本的最终集成和修复 |
| `test` | **测试环境分支**，用于部署给 QA 测试 |

### 正确的操作流程（以 `version` 为 release 分支）

```
1. 从 version 拉开发分支 feature/xxx
   ↓
2. 开发完成，合并回 version（code review + CI 通过）
   ↓
3. 将 version 分支代码部署到 test 环境（不是合并代码，是部署）
   ↓
4. QA 在 test 环境测试
   ↓
5. 测试发现 bug → 在 version 分支修复 → 重新部署 test 环境（反复直到测试通过）
   ↓
6. 测试全部通过后，将 version 分支合并到 master（上线）
   ↓
7. 在 master 上打 tag（如 v2.0.0）
   ↓
8. 将 master 合并回 version（保持同步，避免以后冲突）
```

## 核心区别：合并代码 ≠ 部署代码

**合并代码**是 Git 操作（修改仓库历史）  
**部署代码**是 DevOps 操作（把代码放到服务器上运行）

它们完全是两回事。

---

## 你的问题："version分支部署到test分支，不合并怎么发布？"

答案：**部署不是用 Git 合并来完成的，而是用 CI/CD 工具直接把 version 分支的代码拉取到测试服务器上运行。**

---

## 具体是怎么做的（3种常见方式）

### 方式1：CI/CD 直接部署（最推荐，零合并）

```
开发完成 → 推送代码到 version 分支
         ↓
CI/CD 工具（Jenkins/GitLab CI/GitHub Actions）监听到 version 有推送
         ↓
自动拉取 version 分支的最新代码
         ↓
构建、打包
         ↓
部署到 test 环境服务器
         ↓
QA 访问 test 环境进行测试
```

**关键点：全程没有 Git 合并操作，test 分支的代码树完全不动。**

---

### 方式2：用 Git 部署脚本（手动方式）

在测试服务器上执行：

```bash
# 登录测试服务器
ssh test-server

# 进入项目目录
cd /var/www/myapp

# 拉取 version 分支的最新代码（不是合并，是拉取）
git fetch origin
git checkout version
git pull origin version

# 重启服务
npm run build && pm2 restart app
```

**这里 `git checkout version` 只是切换分支，并没有把 version 合并到 test。**

---

### 方式3：用"部署分支"（你们现在的误区）

有些人会这样做（**不推荐**）：

```bash
git checkout test
git merge version   # 把 version 合并到 test
git push origin test
# 然后 CI 监听 test 分支，部署 test 分支的代码
```

**这种做法的坏处：**
- test 分支的历史被污染（充满合并提交）
- 如果有人在 test 分支上直接改代码（比如加调试日志），这些代码永远不会进 version，但会留在 test 分支里
- 时间长了，test 和 version 分叉，你根本不知道 test 环境跑的是哪个版本的代码

└──────┬──────┘
       │ 合并

┌─────────────┐

          └─────────────┘

## 你们的 CI/CD 应该怎么配置

以 **GitLab CI** 为例（Jenkins 类似）：

```yaml
# .gitlab-ci.yml
deploy-to-test:
  stage: deploy
  script:
    - ssh test-server "cd /app && git pull origin version && npm install && npm run build && pm2 restart app"
  only:
    - version   # 只有 version 分支推送时才触发

deploy-to-stage:
  stage: deploy
  script:
    - ssh stage-server "cd /app && git pull origin version && npm install && npm run build && pm2 restart app"
  only:
    - version
  when: manual  # 手动触发
```

这样，**每次推送到 version 分支，test 环境自动更新，但 test 分支本身永远不会收到合并提交。**

## 一句话总结

> **"version部署到test"的意思是：把 version 分支的代码复制到 test 服务器上运行，而不是用 Git 合并到 test 分支。**  
> 部署是运维操作（拷贝代码+重启服务），合并是代码管理操作（修改仓库历史），两者无关。

如果你们的流程里确实要求"必须先合并到 test 分支才能部署"，那是你们 CI/CD 配置的问题，建议改成**直接监听 version 分支触发部署**，这样更干净。

## 为什么大家老提"MR 到 test"？

因为这是**最直观、最容易理解**的协作方式：

```
开发完成 → 提 MR 到 test → 测试环境自动部署 → QA 测试 → 提 MR 到 master → 上线
```

这种模式在**小型团队、快速迭代**的项目中非常普遍，因为它：
- **简单粗暴**，新人一看就懂
- CI/CD 配置简单（监听 test 分支触发部署）
- 代码审核（MR）和部署绑定在一起

## 但是！这种模式有 3 个致命问题

### 问题 1：test 分支变成"垃圾场"

```
test 分支的历史：
- feat/A 合并到 test（带调试日志）
- fix/B 合并到 test（带临时配置）
- feat/C 合并到 test（冲突解决时的临时提交）
- 回滚某次提交
- 再次合并...
```

**结果**：test 分支的历史一团糟，你根本看不清"当前测试环境到底包含了哪些功能"。

---

### 问题 2：测试和上线脱节

```
情况：
1. feat/A 合并到 test，QA 开始测试
2. feat/B 也合并到 test（因为等不及了）
3. QA 发现 bug，但不知道是 A 还是 B 引入的
4. 紧急修复后，直接合并到 test
5. 测试通过，从 test 合并到 master 上线

问题：test 上可能包含临时调试代码、未完成的功能、甚至只有测试用的 mock 数据
这些全被带到 master 上了！
```

---

### 问题 3：无法做"版本锁定"

标准流程应该是：

```
version 分支 = v2.0.0 版本的所有代码（锁定版本）
test 环境 = 部署 version 分支的代码（测试这个版本）
```

但如果用"MR 到 test"：
```
test 分支 = 不断变化的"大杂烩"
你没法说"我正在测试 v2.0.0 版本"
因为 test 上可能混入了 v2.1.0 的代码
```

## 两种模式的本质区别（重点！）

| 维度 | MR 到 test | version 部署到 test |
|------|-----------|-------------------|
| **test 分支** | 代码的"集散地" | 只是"部署目标"，不存代码 |
| **部署触发** | 合并到 test 触发 | 推送 version 触发 |
| **版本概念** | 没有版本，持续集成 | 锁定版本（release/2.0.0） |
| **上线方式** | test → master（容易带垃圾） | version → master（纯净） |
| **适合场景** | 持续交付（CD） | 版本发布（Release） |

## 正确的并行测试方案（大厂标准做法）

### 方案 1：每个 version 独立部署一套测试环境（推荐）

```
version/2.0.0 → 部署到 test-2.0.0 环境 → QA 测试
version/2.1.0 → 部署到 test-2.1.0 环境 → QA 测试
version/2.2.0 → 部署到 test-2.2.0 环境 → QA 测试

每个版本有独立的测试环境，互不干扰
测试通过后，按顺序合并到 master 上线
```

**如何实现：**
```yaml
# GitLab CI 示例
deploy-test:
  script:
    - 部署到 test-${CI_COMMIT_REF_NAME}  # 用分支名命名环境
  only:
    - /^version\/.*$/  # 匹配 version/ 开头的分支
```

**优点：**
- 三个版本可以并行测试
- 互不干扰
- 测试通过后随时可以上线

**缺点：**
- 需要多套测试环境（云资源成本）
- 环境管理复杂

---

### 方案 2：功能开关（Feature Flag）+ 单一 test 环境

```
所有 version 都部署到同一个 test 环境
但用 Feature Flag 控制功能是否生效：

test 环境同时包含：
- 2.0.0 的支付重构（默认关闭，QA 单独开启测试）
- 2.1.0 的推荐算法（默认关闭，QA 单独开启测试）
- 2.2.0 的 UI 改版（默认关闭，QA 单独开启测试）

QA 测试时：
- 测 2.0.0 → 只开启 2.0.0 的 flag
- 测 2.1.0 → 只开启 2.1.0 的 flag
- 测 2.2.0 → 只开启 2.2.0 的 flag
```

**实现方式：**
```javascript
// 代码中
if (featureFlags.isEnabled('payment_v2')) {
  // 新版支付逻辑
} else {
  // 旧版支付逻辑
}
```

**优点：**
- 只需一套 test 环境
- 可以并行测试多个版本
- 上线时只需开启 flag，无需重新部署

**缺点：**
- 需要引入 Feature Flag 系统（如 LaunchDarkly、Unleash）
- 代码中会积累大量 if/else，增加复杂度
- 需要管理 flag 的生命周期（测试完要清理）
